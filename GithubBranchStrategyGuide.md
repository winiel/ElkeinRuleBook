# 프로젝트 브랜치 전략 및 작업 워크플로우

본 프로젝트는 안정적인 서비스 운영과 안전한 기능 개발을 위해 `main` 브랜치와 `dev` 브랜치를 엄격하게 분리하여 운영합니다.

## 🌟 기본 원칙
* **main 브런치**: 테스트와 검증이 완전히 끝난, 언제든 배포 가능한 상태의 코드만 존재해야 합니다. 직접적인 수정은 엄격히 금지됩니다.
* **dev 브런치**: 새로운 기능 추가, 버그 수정 등 모든 실질적인 개발 작업이 이루어지는 공간입니다.

---

## 💻 단계별 작업 지침 (Terminal 명령어)

### 1단계: dev 브랜치 생성 및 이동 (프로젝트 초기 1회)
최초에 `main` 브랜치에서 파생된 개발 전용 `dev` 브랜치를 만듭니다.

~~~
bash
# 1. 최신 main 브랜치 상태인지 확인합니다.
git checkout main
git pull origin main

# 2. dev 브랜치를 생성하고, 해당 브랜치로 작업 환경을 이동합니다.
git checkout -b dev
~~~

### 2단계: dev 브랜치에서 기능 개발 및 저장 (Commit & Push)
코드를 수정하고 기능 보완을 진행한 뒤, 변경 사항을 dev 브랜치에 안전하게 저장합니다.

~~~
# (코드 작성 및 수정 작업 진행...)

# 1. 변경된 모든 파일을 장바구니(Staging Area)에 담습니다.
git add .

# 2. 어떤 기능을 추가/보완했는지 명확한 메시지와 함께 버전을 기록합니다.
git commit -m "feat: 신규 테이블 스키마 추가 및 로직 보완"

# 3. 내 컴퓨터의 dev 브랜치 작업 내역을 Github 서버의 dev 브랜치로 밀어 올립니다.
# (최초 push 시에만 -u 옵션 사용, 이후에는 git push origin dev만 입력)
git push -u origin dev
~~~

### 3단계: 테스트 및 검증
Github 서버에 올라간 dev 브랜치의 코드가 정상적으로 작동하는지, 버그는 없는지 충분히 테스트를 진행합니다. (이 단계는 코드 명령어가 아닌 실제 서비스 테스트 단계입니다.)

### 4단계: 검증 완료 후 main으로 병합 (Merge)
모든 테스트가 무사히 완료되었다면, 이제 dev의 완성된 코드를 main으로 합칩니다.

~~~
# 1. 병합을 위해 기준이 되는 main 브랜치로 이동합니다.
git checkout main

# 2. 혹시 모를 원격 저장소의 최신 main 변경사항을 미리 받아옵니다.
git pull origin main

# 3. 검증이 끝난 dev 브랜치의 내용을 main 브랜치로 병합합니다.
git merge dev

# 4. 병합되어 업데이트된 최종 main 브랜치를 Github 서버에 반영합니다.
git push origin main
~~~
